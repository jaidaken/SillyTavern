## Deferred major refactor: reactive model (parked, confirmed real R&D)

Today the whole app is ONE client-rendered component (ChatView) driven by a module-global var +
`zx.client.rerender()`, so every state change (open a panel, every streamed token) rebuilds the entire
vdom. ziex's scoped primitive (`State(T)` + `scheduleRender(component_id)`, `.ziex/.../reactivity.zig`)
is component-local and keyed by generated ids; there is a `component_subscriptions` map but its notify
path looks unfinished, and `scheduleRender` on an unregistered nested component falls back to
`renderAll()`. So the clean win is not "flip to ctx.state" but DECOMPOSING into registered CSR
components. Staged plan agreed with operator:

1. Make each message + the message-log their own `@rendering={.client}` components, so a streamed token
   re-renders only the streaming message, not all N. Build it, then MEASURE re-render cost before/after.
2. If that holds, decompose the shell (topbar / panel host / composer) into CSR components too, so panel
   toggles stop re-rendering the log.

Risk: ziex is alpha; many registered components + nested scheduleRender are unproven at this scale, and
the shared-state subscription path is not obviously finished. Measure before committing the full pass.
This refactor changes component structure, so it lands BEFORE the colour feature and the a11y markup
work (both touch the same components) to avoid double-work.

### PROBE RESULT 2026-07-10: MAJOR (premise does not hold on ziex @ 26f5945)

A disposable design-probe (member p1probe, worktree reverted) built the scoped-render slice, measured it,
and reverted. Verdict MAJOR: per-message `@rendering={.client}` is NOT viable, and no variant of it is,
on this ziex. Lead-verified the load-bearing claims against `.ziex/src`:

- `rerender()` (`reactivity.zig:204-208`) unconditionally calls `renderAll()`; it is the ONLY public
  reactive entry point (`scheduleRender` is a private const, only `rerender` + `State` are re-exported).
- `scheduleRender` (`reactivity.zig:214-227`) falls back to `renderAll()` for any component not in the
  compile-time registry (every nested-in-CSR component), per its own comment.
- `@rendering={.client}` is a NO-OP on the client build (`x.zig:543-545`): a nested client component is
  inlined into the parent vtree, no independent render boundary.
- component ids are per-JSX-SITE not per-instance (`Transpile.zig` `generateComponentIdInner` = md5 of
  name+path+site-index): a `{for}` loop of N messages shares ONE id, N vtrees collapse to one slot.
- `component_subscriptions` (`reactivity.zig:31-36`) is a STUB, never inserted or read.
- `store.Message` cannot even be a client-component prop: `@rendering={.client}` routes props through
  zxon which yields `[]const u8`, a hard compile error against `Message.{name,body}_owned: ?[]u8`.

MEASUREMENT (11 msgs, 5 tokens, render-fn counters through the wasm door): every token re-renders all 11
messages in all three variants (11.00 MessageView calls per ChatView render, byte-identical before/after).
BUT the 11x is FUNCTION invocations, not DOM writes: the vdom diff suppresses most DOM mutation and the
`sanitized.zig` / `html.zig` render cache already skips re-sanitizing unchanged bodies (16 sanitizes for 88
MessageView calls). DOM-patch cost was NOT measured, so the user-visible cost is bounded above by these
numbers and may already be small. No evidence yet that the single-component render is too slow.

### DESIGNED PLAN 2026-07-10 (after a full deep-read of the ziex render/reactivity/diff/transpile core + the app layer)

Read into context: `.ziex/src` {x.zig, runtime/client/reactivity.zig, runtime/client/Client.zig,
runtime/client/render.zig, runtime/server/render.zig, runtime/client/window/document.zig,
runtime/core/vdom.zig, root.zig, runtime/client.zig, runtime/core/App/Client.zig, core/Transpile.zig
id-gen + client-registry emit}; app {store, stream, sanitized, html, ui_state, ui, all .zx, build.zig}.

REFRAME (the number that changes the plan): ziex's diff is ALREADY granular at the DOM level. On a
streaming token only the streaming message's `mes_text` differs, so the diff emits ONE `RAW_HTML` patch
(`vdom.zig:283`); unchanged messages produce no patch; the shell/composer produce no patch. So DOM writes
per token are ~1 regardless of chat length. The real cost is CPU: `reconcileChildren` (`vdom.zig:465`)
eagerly `resolveComponent`s EVERY child before diffing, so all N `MessageView` fns run and all N mes_text
bodies get `concatRawText` + `eql` every token. That is O(sum of message sizes) per token, and it grows
with conversation length + per-message richness (colour, avatars, markdown). The scaling concern is real;
"measure first / maybe it is fine" was wrong because the cost provably grows as we build.

THE FIX (both in Zig/ziex, no new JS, no glue-side DOM hack):
1. MEMOIZATION (the actual perf fix; React.memo equivalent). Teach `reconcileChildren` to skip a child
   whose identity + content are unchanged BEFORE it calls `resolveComponent` (so `MessageView` is not even
   invoked for a sealed message). Sealed message bodies are immutable + store-owned with a STABLE pointer
   (`store.zig` endStream), so the skip key is cheap and correct: same message key + same body pointer =
   skip. Turns per-token work from all-N-messages to just-the-streaming-one. Small, targeted ziex change
   (a few lines in `reconcileChildren` + a `key`/memo signal from the app). This is patch #5.
2. REGION DECOMPOSITION (code structure + cross-region scoping; the audit's original intent). Split the
   monolithic ChatView into top-level `@rendering={.client}` components: Shell (topbar+drawers), MessageLog,
   Composer, SidePanel docks. `renderAll` (`Client.zig:301`) iterates the compile-time registry and
   `scheduleRender(id)` (`reactivity.zig:214`) renders exactly one registered entry, so each region becomes
   independently re-renderable. Convert mutations from the module-global-var + `rerender()` pattern to ziex
   `State` (which calls `scheduleRender(owning-component-id)` on `.set`, `reactivity.zig:100`), so a panel
   toggle / composer keystroke / settings change / resize scopes to its region and never touches MessageLog.
   The door (bridge.zig) drives MessageLog by a State handle the component publishes to a module global on
   first render (or a small `scheduleRenderByName` ziex helper). No new marker machinery needed: these are
   static top-level sites the transpiler already emits markers + registry entries for.

CONSIDERED AND REJECTED: rebuild ziex to mount each message as a fully independent RUNTIME component
instance (runtime marker mint + a runtime-extensible instance registry + per-instance ids + real
client-side CSR mounting, which today is a stub empty `<div>` at `render.zig:310-316`). It is the
maximally-complete version, but it is major surgery on the framework core for NO behavioral benefit over
memoization: the streaming message is the only one that changes token-to-token, and memoization already
isolates it. Reserve it only if the design-probe shows memoization cannot be made to fire correctly.
REJECTED also: glue writes the streaming tail straight to the DOM (adds JS, bypasses the framework).

## PLAN: order to tackle everything

Rationale (LOCKED 2026-07-10 with operator): bugs first for a clean verified baseline (they sit in
disjoint code from the rewrite - glue JS / Zig backend / CSS, not the component/reactive layer - so
fixing them first is near-zero rework, and any rewrite regression is then provably the rewrite's). The
rewrite goes SECOND, early, on that clean baseline, because it reshapes components and everything after
(colour, a11y, component nits) must build on the final shapes, not shapes we are about to change. Then
cleanup, splits, features, polish, a11y, perf, verify.

- **Phase 0 - Bugs.** The 8 real defects (instantiateStreaming brick, class-remap whitespace, stream
  alloc-throw stall, appendMessage leak, resize-drag leak, composer focus ring, [DONE] non-termination,
  LATEXMATHSPANS). Correctness + unblocks real streaming. Test where testable.
- **Phase 1 - Reactive-model rewrite (the deferred item). RE-DESIGNED 2026-07-10 after the probe MAJOR +
  the deep read (see DESIGNED PLAN above).** Not a per-message client-component rewrite (not viable on this
  ziex). Instead: ziex memoization patch (#5) so the diff skips unchanged messages = the actual per-token
  cost fix + region decomposition (Shell/MessageLog/Composer/SidePanel as top-level client components) with
  State-scoped mutations. W0 fixes the regression gate first; W1 design-probes the memoization slice; W2
  builds it. Sets the component shapes the later phases build on, so it stays first. Awaiting operator go.
- **Phase 2 - Dead code + cheap nits.** Probe/dev block out of prod glue, always-on stats, `window.*`
  globals, dead tokens/rules, motion single-owner, batched Zig nits, dead CSS. Component-touching nits now
  land on the post-rewrite shapes.
- **Phase 3 - Structure splits.** app.css into token/base/component layers; main.js into modules
  (wasm-bridge / sanitize / highlight / stream / ui-glue, probes gone). main.js needs module infra
  (type=module or dynamic import) - a design decision to make here.
- **Phase 4 - Per-speaker colour feature.** Data model + rendering on the new component shapes; user
  persona colour + optional character colour; fixtures demo. Picker UI waits on persona/character panels.
- **Phase 5 - Design system + CSS polish.** On the split CSS: modular type scale, tokenise strays

- **Phase 6 - Accessibility.** Motion radiogroup, drawer `aria-expanded`/`aria-controls` + `type=button`,

## Discovered during Phase 0 (fold into later phases)

- glue-side [DONE] latch [Phase 3, glue stream module]: stream.zig now seals on [DONE], but the glue's
  `streamActive` stays true until the socket closes, so a backend that holds the socket open after
  [DONE] still blocks a later `startStream` from the JS side. Fix when restructuring the stream module:
  add an export the glue polls (stream reached .done) and cancel the reader on it. The Zig state machine
  is correct; the JS one is not yet.
- glue/main.js `env.sanitize` returns `writeString(...)` and never frees on a wasm-side throw [Phase 3].
- glue/main.js `readString` decodes off `wasm.memory.buffer` each call; a memory growth between the
  `__zx_alloc` and the read would detach the view [Phase 3].
- `boot()`'s probe block appends `#probe-metrics` twice if both `growth` and `stream` params are set
  (duplicate element id) [Phase 2, dead/probe code removal].
